`typed-xlsx` is a TypeScript-first library for generating XLSX files. It ships with its own OOXML serialization and ZIP assembly engine — no SheetJS dependency, no wrapper around another library's limitations.

## What problems does it solve?

### Excel generation has a low ceiling in most libraries

Most Excel libraries for Node.js give you a row/cell model that gets unwieldy fast. Once you need multi-row expansion from arrays, column summaries, per-row conditional styling, freeze panes, and multi-table sheet layouts, you end up building a framework on top of a framework. `typed-xlsx` provides that layer as a first-class public API.

### Streaming was an afterthought everywhere else

Other libraries advertise streaming but still materialize the full workbook before writing. `typed-xlsx` has two separate builders: a buffered one for normal workbooks and a streaming one that serializes incrementally and pipes directly into the ZIP writer. Peak memory scales with your batch size, not your dataset size.

### Schemas need to be reusable across different export shapes

A common pattern is one canonical schema used for multiple exports — full data, filtered columns, different audience. The builder system (`createExcelSchema()`) produces a plain definition object that carries no state, so it can be passed to any number of tables across multiple workbooks without side effects.

## What you get

- **Typed accessor layer** — dot-path strings (`"user.address.city"`) and callback functions (`row => ...`) both resolve with full TypeScript inference against your row type.
- **Reducer-based summaries** — summaries accumulate row by row with `init` / `step` / `finalize`. The same schema definition works in both buffered and stream exports.
- **Library-owned `CellStyle`** — styles are defined with the library's own normalized type, independent of any backend serializer.
- **Custom OOXML + ZIP engine** — the output pipeline writes raw XML and assembles the ZIP archive without any third-party spreadsheet library in the call stack.
- **Real streaming** — `createWorkbookStream()` writes each sheet incrementally to a spool (file or memory), then pipes the ZIP archive to any Node writable, Web writable, file, or readable stream.
- **Multi-table layouts** — multiple tables per sheet with configurable column/row gaps and grid placement.
- **Freeze panes and RTL sheets** — first-class sheet view options for both builders.

## What the custom OOXML writer unlocks next

Owning the XLSX writer means the roadmap is no longer limited by what a wrapper library happens to expose. For the kind of workbook `typed-xlsx` already excels at — operational, financial, audit, and management reports — the most useful next Excel-native features are the ones that make generated workbooks feel more native inside Excel itself.

The examples below are **illustrative API sketches, not implemented features**. They show the kind of ergonomics that would fit naturally beside existing options such as `freezePane`.

### 1. Excel tables with filters and totals

Useful for: financial statements, audit extracts, reconciliation reports, and any export users immediately want to filter/sort in Excel.

```ts
const workbook = createWorkbook();

workbook.sheet("Revenue").table({
  id: "revenue",
  rows,
  schema,
  excelTable: {
    name: "RevenueTable",
    style: "TableStyleMedium2",
    autoFilter: true,
    totalsRow: true,
  },
});
```

Why it matters:

- end users get native filter dropdowns immediately
- totals can stay visible at the bottom of the table
- formulas can reference columns by name instead of fragile `B2:B5000` ranges

### 2. Formula columns and cross-sheet KPIs

Useful for: variance reports, margin analysis, rolling totals, budget vs actual, and board-pack summary sheets.

```ts
const schema = createExcelSchema<Row>()
  .column("target", { header: "Target", accessor: "target" })
  .column("actual", { header: "Actual", accessor: "actual" })
  .column("variancePct", {
    header: "Variance %",
    formula: ({ row }) => `IFERROR((${row.actual}-${row.target})/${row.target},0)`,
    style: { numFmt: "0.0%" },
  })
  .build();
```

```ts
workbook.sheet("Summary").cells([
  {
    cell: "B2",
    value: { formula: "=SUM(Revenue!F:F)" },
    style: { numFmt: "$#,##0.00" },
  },
]);
```

Why it matters:

- workbook values can recalculate inside Excel
- users can audit formulas directly
- summary sheets can stay linked to detailed tabs

### 3. Conditional formatting for report scanning

Useful for: SLA breaches, overdue items, negative margins, risk flags, and traffic-light dashboards.

```ts
workbook.sheet("KPIs")
  .table({ id: "kpis", rows, schema })
  .conditionalFormatting({
    range: "D2:D200",
    rules: [
      {
        type: "iconSet",
        iconSet: "3TrafficLights1",
        thresholds: [0.9, 1],
      },
    ],
  });
```

Why it matters:

- the report becomes scannable without manual styling
- thresholds live in the workbook instead of being baked into static colors
- users can change values and still keep the visual signals

### 4. Hyperlinks and sheet navigation

Useful for: multi-sheet report packs, audit exports, drill-down workflows, and linking back to internal tools.

```ts
const schema = createExcelSchema<Order>()
  .column("orderId", {
    header: "Order",
    accessor: "id",
    hyperlink: (row) => ({
      location: `https://internal.app/orders/${row.id}`,
      label: row.id,
    }),
  })
  .build();
```

```ts
workbook.sheet("Summary").cells([
  {
    cell: "B4",
    value: "Open overdue invoices",
    hyperlink: { sheet: "Invoices", cell: "A1" },
  },
]);
```

Why it matters:

- summary tabs can jump straight to detail sections
- exported reports can connect back to source systems
- large workbooks become easier to navigate

### 5. Print setup and repeating headers

Useful for: board packs, finance reviews, printed audit workpapers, and PDF exports sent to stakeholders.

```ts
workbook.sheet("Board Pack", {
  freezePane: { rows: 2 },
  printSetup: {
    orientation: "landscape",
    fitToWidth: 1,
    repeatRows: { from: 1, to: 2 },
    margins: "narrow",
    headerFooter: {
      footer: "Page &[Page] of &[Pages]",
    },
  },
});
```

Why it matters:

- printed output keeps column headings on every page
- wide reports can fit predictably
- generated workbooks become presentation-ready without manual cleanup

### 6. Protection and grouped detail sections

Useful for: controlled templates, governed reports, and packs where executives need summary-first views with optional drill-down.

```ts
workbook.sheet("Forecast", {
  protection: {
    sheet: true,
    selectUnlockedCells: true,
  },
  outline: {
    summaryRowsBelow: false,
  },
});
```

```ts
workbook.sheet("Forecast")
  .table({ id: "forecast", rows, schema })
  .groupRows([
    { from: 2, to: 20, level: 1, collapsed: true },
    { from: 21, to: 60, level: 1, collapsed: true },
  ]);
```

Why it matters:

- formulas can stay locked while assumption cells remain editable
- reports can open in an executive-summary state
- detailed backup sections stay available without overwhelming first-time readers

If the goal is to prioritize like-for-like additions next to `freezePane`, the strongest candidates are probably:

1. Excel tables
2. formulas
3. conditional formatting
4. hyperlinks
5. print setup

More advanced features such as charts, images, pivot tables, slicers, threaded comments, and embedded objects are also unlocked by owning the OOXML layer, but they are likely a second wave after the core report-focused features above.

## When to use `typed-xlsx`

Use `typed-xlsx` when your data has a known structure and your reports need consistent column definitions, summaries, and styling. It is not designed for fully dynamic layouts where the row/column structure is unknown at code time.

| Scenario                                                   | Recommendation                             |
| ---------------------------------------------------------- | ------------------------------------------ |
| Regular reports from typed data models                     | `typed-xlsx`                               |
| Streaming exports of 100k+ rows                            | `typed-xlsx` with `createWorkbookStream()` |
| Fully dynamic workbook where shape is runtime-determined   | Lower-level library                        |
| Reusing one schema definition across multiple export views | `typed-xlsx`                               |
